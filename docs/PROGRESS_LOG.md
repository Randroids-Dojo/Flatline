## 2026-05-05, Hitstop spares the touch-look camera (PR #64 review fix)

- Branch: `feat/hitstop-on-hit`
- PR: #64
- Changed: split the per-frame delta in `src/components/FlatlineGame.tsx` so hitstop only slows the simulation, not the touch-look camera. The earlier hitstop slice computed a single `delta = rawDeltaSeconds * hitstopScale` and used it for both sim consumers and touch-look yaw/pitch, which meant the camera also slowed during the 30 to 60 ms freeze window. Desktop mouse-look was already on wall-clock (it reads `event.movementX/Y` directly inside the keyboard / mouse effect), so that path is unaffected.

## 2026-05-05, Mobile joystick uses TouchEvent to bypass stale PointerEvent coords

- Branch: `feat/hitstop-on-hit`
- PR: #TBD
- Changed: switched the mobile joystick handlers in `src/components/FlatlineGame.tsx` from `PointerEvent` (`onPointerDown` / `onPointerMove` / `onPointerUp`) to `TouchEvent` (`onTouchStart` / `onTouchMove` / `onTouchEnd`). Real-device confirmation across two preview slices: even after dropping the `(0, 0)` rejection guards and adding a snap-on-move helper, the move stick on the user's Android Chrome session was still anchored at the top-left corner. Backgrounding and foregrounding the tab unstuck it. The signature matches the documented Android Chrome bug where `PointerEvent.clientX/Y` inherits from a stale `MouseEvent` baseline until visibility changes force the browser to refresh pointer state, while `TouchEvent.changedTouches[i].clientX/Y` reads directly from the OS-level Touch object and is unaffected. Each new handler iterates `event.changedTouches` and uses `touch.identifier` where the prior handler used `event.pointerId`. Click synthesis on menu buttons is preserved by the same rule from PR #60: `preventDefault()` is called on `touchstart` only when `beginTouch` claims a half-screen joystick, on `touchmove` only when the touch is one of the active joysticks, and never on `touchend`. The desktop input paths (mouse-locked `mousemove`, `mousedown` for fire) live in a separate effect and are untouched. The `dispatchTouch` Playwright helper in `tests/smoke.spec.ts` now constructs a real `Touch` object and dispatches `TouchEvent` so the existing synthetic-touch e2e coverage continues to exercise the active code path.
- Verification: dash check (clean), `git diff --check` (clean), `npm run typecheck`, `npm run lint`, `npm run test` (29 files / 208 tests pass), `npm run build` (success), `npm run test:e2e` (9 passed, 3 skipped).
- Assumptions: Recommended default replaces the pointer joystick path entirely with the touch path because there is no real-world Flatline player on a non-touch input that needs joystick activation; mouse and pen users on desktop are served by the keyboard plus locked-mouse-look path that lives in a separate effect. Recommended default keeps the existing `releaseAllTouches` on `blur` and `visibilitychange:hidden` because those listeners fixed a different real-device bug (stuck stick visuals after `pointercancel` was stolen by the OS gesture system) and removing them would regress that fix. Recommended default keeps `preventDefault` off `touchend` per the lesson recorded in `docs/PROGRESS_LOG.md` 2026-05-04 (mobile menu touch fix) so menu button clicks still synthesize.
